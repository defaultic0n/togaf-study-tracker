Phase A — Objectives

Phase A sets the entire context for the architecture engagement. There are three core objectives:

1. Establish the architecture project

Secure formal recognition and approval to proceed with the architecture work.

2. Develop Architecture Vision

Create a high-level aspirational view of the target architecture aligned to business goals and stakeholder concerns.

3. Get approval via Statement of Architecture Work

Produce a Statement of Architecture Work (SAW) that formally describes the scope, approach, and schedule — and secure sign-off.

Scoping the
Architecture

Step 6 of Phase A is Define Scope. TOGAF defines four dimensions of scope — and the exam tests all four. Getting this wrong leads to architects duplicating work or leaving gaps.

★ Why this matters in the exam

Questions often describe an architecture scenario and ask which scoping dimension is being addressed, or ask you to identify which dimension is missing from a given scope statement.

Dimension 1: Breadth

Breadth defines the horizontal extent of the architecture — which parts of the enterprise are covered.

The core question

What is the full extent of the enterprise, and what part of that extent will this architecture effort deal with?

Why it's not simple

  • Many enterprises span multiple business units that could each be considered enterprises in their own right
  • Modern enterprises extend beyond traditional boundaries — including suppliers, partners, and customers
  • Breadth must be set to reflect the organisational authority of the architecture team
★ Exam tip

Breadth answers "how wide?" — horizontally across the organisation. Narrowing breadth = focusing on one division, region, or business unit rather than the whole enterprise.

8 / 20

Dimension 2: Depth

Depth defines the level of detail the architecture will descend to — and where it stops before becoming system design or engineering.

The core question

To what level of detail should the architecture go? How much architecture is enough? Where is the line between architecture and system design?

Abstraction levels

  • Contextual — strategic overview, whole enterprise
  • Conceptual — domain-level concepts, no technology specifics
  • Logical — functional components, platform-independent
  • Physical — specific products, configurations
★ Exam tip

Depth answers "how deep?" — how much detail. A strategic architecture is shallow; a solution architecture is deep. The ADM can be applied at any depth level.

9 / 20

Dimension 3: Time Period

Time Period defines the planning horizon for the Architecture Vision and how many Transition Architectures will be needed.

The core question

What time period needs to be articulated for the Architecture Vision? Does the same time period apply to the detailed Architecture Description?

Transition Architectures

If the target is far away, the team defines Transition Architectures — intermediate stable states on the path to the target. The number of Transition Architectures and their time periods are scoping decisions made in Phase A.

★ Exam tip

Time Period answers "how far ahead?" — typically a 3-5 year horizon for enterprise architecture. Transition Architectures are how you bridge the gap between now and that horizon.

10 / 20

Dimension 4: Architecture Domains

A complete Enterprise Architecture covers all four domains — but resource constraints mean you often cannot do all four at once.

Business

Phase B — strategy, capabilities, processes, organisation

Data

Phase C — data entities, data management, information flows

Application

Phase C — logical and physical applications, interactions

Technology

Phase D — platforms, infrastructure, standards

★ Exam tip

Domains answer "which architecture types?" A Technology-only architecture is a valid scoped engagement. TOGAF says breadth, depth, and time are determined first — then suitable domains are selected to match.

Section 3 of 5 · Phase A Outputs

Phase A — Key Outputs

The primary output is the Statement of Architecture Work — but Phase A also produces several other key artefacts:

Statement of Architecture Work (SAW)

Defines scope, approach, schedule, and resource requirements. Formally approved by sponsors. This is the contract for the architecture engagement.

Architecture Vision

A high-level description of the target architecture — a summary view of what the future state looks like, without full detail.

Refined Architecture Principles

Principles confirmed or updated for this specific engagement.

Communications Plan

How architecture information will be shared with stakeholders throughout the engagement.

13 / 20

Statement of Architecture Work

The SAW is the most important output of Phase A. It is the formal document that authorises the architecture work to proceed.

Typical contents of the SAW

  • Title and scope of the architecture work
  • Business context and drivers
  • Architecture Vision (high-level)
  • Proposed approach, including which ADM phases will be used
  • Roles, responsibilities, and governance arrangements
  • Schedule and resource plan
  • Risk and assumptions
  • Request for Architecture Work reference (the trigger)
★ Exam tip

The SAW is the output of Phase A. The Request for Architecture Work is the input that triggers Phase A. Don't confuse these two documents.

14 / 20

The Architecture Vision Document

The Architecture Vision is a concise view of the target enterprise architecture — created to agree direction before the full detailed architecture is developed.

What it contains

  • Problem description
  • Objective of the engagement
  • Summary of baseline and target
  • Stakeholder map
  • Key concerns addressed

Why it exists

Stakeholders need a clear, accessible summary before committing to a full architecture engagement. The Vision gets early alignment without the cost of full Architecture Definition.

★ Exam tip

The Architecture Vision is a deliverable (a formal work product). It is also the name of Phase A itself. In exam questions, context matters — "Architecture Vision" could mean the phase OR the document.

Section 4 of 5 · Stakeholders & Concerns

Stakeholders & Concerns in Phase A

Identifying and addressing stakeholder concerns is a core activity of Phase A. The architecture exists to address these concerns — not to demonstrate technical elegance.

Stakeholder concerns drive architecture

  • Each stakeholder type has different concerns
  • Architecture Views are created to address specific concerns
  • Unaddressed concerns = architecture gaps

Tools used in Phase A

  • Stakeholder map (power/interest grid)
  • Concerns catalogue
  • Communications Plan
  • Architecture Vision document
16 / 20

Concerns → Viewpoints → Views

TOGAF uses a structured approach to ensure architecture addresses what matters to stakeholders:

1
Concern — what a stakeholder cares about (e.g., "how does this affect security?")
2
Viewpoint — a convention for constructing and interpreting a view addressing that concern (e.g., Security Viewpoint)
3
View — the actual representation of the architecture from that perspective (e.g., Security Architecture diagram)
★ Exam tip

A viewpoint is the template/pattern. A view is the instance created from that template for a specific architecture. This distinction is frequently tested.

Section 5 of 5 · Exam Focus

Common Phase A
Exam Traps

These distinctions come up repeatedly in Foundation exam questions:

Request for Architecture Work ≠ Statement of Architecture Work

The Request is the INPUT to Phase A (the trigger). The Statement is the OUTPUT of Phase A (the authorisation). Don't swap them.

Architecture Vision ≠ Architecture Definition Document

Vision = high-level summary from Phase A. Architecture Definition Document = detailed output built across Phases B, C, D.

Viewpoint ≠ View

Viewpoint is the pattern/template. View is the actual diagram or description created using that template.

18 / 20

Scoping — Exam Traps

Trap 1: Confusing Breadth and Domains

"We will only address IT systems, not business processes" — this is a Domains decision (excluding Business domain), not a Breadth decision. Breadth = which part of the enterprise. Domains = which architecture types.

Trap 2: Forgetting the Time Period

Questions about Transition Architectures often hinge on Time Period scoping. The number and timing of Transition Architectures is decided in Phase A, not Phase E.

Trap 3: Depth vs Detail Phase

Depth is set in Phase A as part of scoping. It doesn't mean the detailed descriptions are done in Phase A — they're done in Phases B–D. Phase A just decides how deep those later phases will go.
