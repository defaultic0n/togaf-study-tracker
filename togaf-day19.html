01

ADM Phases — Quick Recap (All 10)

The full cycle

Remember: Preliminary + A–H + Requirements Management. The exam frequently tests which deliverable belongs to which phase, what triggers which phase, and the purpose of each phase.

Prelim
Set up the architecture capability
Tailored framework · Principles · Architecture team established
Phase A
Architecture Vision — scope and approve
Key output: Statement of Architecture Work · Architecture Vision · Stakeholder Map · Communications Plan
Phase B
Business Architecture — current + target + gap
Key output: ADD Business chapter · Org maps · Business Interaction Matrix · Gap Analysis
Phase C
IS Architecture — data + applications
Key output: ADD IS chapter · App Interaction Matrix · Data Dissemination Diagram · Gap Analysis
Phase D
Technology Architecture — platform services
Key output: ADD Technology chapter · Technology Portfolio Catalogue · Gap Analysis · Uses TRM
Phase E
Opportunities & Solutions — first roadmap + work packages
Key output: Architecture Roadmap (initial) · Transition Architectures · ABB→SBB mapping
Phase F
Migration Planning — finalise plan
Key output: Implementation & Migration Plan · Architecture Roadmap (final) · Architecture Contracts (draft)
Phase G
Implementation Governance — oversight
Key output: Signed Architecture Contracts · Compliance Assessments · Governance Log entries
Phase H
Architecture Change Management — respond to change
Key output: Architecture Change Requests · Updated ADD · Change impact assessments
Req Mgmt
Only continuous phase — runs throughout all phases
Key output: Architecture Requirements Specification (continuously updated)
⭐ Exam tip

The Architecture Roadmap first appears in Phase E — never before. The Architecture Definition Document grows across Phases B, C, D. The Statement of Architecture Work is Phase A's primary output. Architecture Contracts are drafted in F, signed in G.

02

Content Framework: Deliverables, Artefacts, BBs

Deliverable
Contractually specified, formally reviewed and approved work product. Examples: Architecture Definition Document, Statement of Architecture Work, Architecture Contract, Architecture Vision.
Artefact
An architectural work product that describes aspects of the architecture. Contained within deliverables. Examples: Org Map, App Interaction Matrix, Data Dissemination Diagram, Technology Portfolio Catalogue.
Architecture Building Block (ABB)
Technology-independent. Defines what a capability does. Created in Phases B, C, D. Lives in Architecture Continuum. Example: "Customer Management capability."
Solution Building Block (SBB)
Technology-specific product. Implements an ABB. Mapped in Phase E. Lives in Solutions Continuum. Example: "Salesforce Sales Cloud."
Content Metamodel
Defines entity types (Actor, Business Service, Data Entity, App Component, Tech Component) and relationships between them. The vocabulary of architecture. Enables traceability from business goals to technology.
⭐ Exam tip

The most-tested Content Framework distinction: Deliverable vs Artefact vs Building Block. Remember: Artefacts are components of Deliverables. ABBs are in the Architecture Continuum; SBBs in the Solutions Continuum. ABB→SBB mapping happens in Phase E.

03

Enterprise Continuum & Repository Recap

Enterprise Continuum — 4 levels (most generic → most specific)

Foundation Architecture (TRM, III-RM — applies to all) → Common Systems Architecture (cross-enterprise patterns) → Industry Architecture (sector-specific) → Organisation-Specific Architecture (your enterprise). Two parallel tracks: Architecture Continuum (ABBs) and Solutions Continuum (SBBs).

Architecture Repository — 6 components
1. Architecture Metamodel · 2. Architecture Landscape (Strategic/Segment/Capability) · 3. Reference Library · 4. Standards Library · 5. Governance Log · 6. Architecture Requirements Repository
Standards Library role
Stores approved technology products, versions, configs. Constrains Phase D technology choices. Non-standard = dispensation required from Architecture Board.
Governance Log role
Audit trail of ALL governance decisions: compliance assessment outcomes, dispensations, change request decisions, risk acceptances. Records every formal Architecture Board decision.
Architecture Landscape — 3 levels
Strategic (whole enterprise, C-suite, 3–5yr) · Segment (business unit, 1–3yr) · Capability (specific project, 0–18 months)
⭐ Exam tip

Architecture Continuum = descriptions/ABBs. Solutions Continuum = implementations/SBBs. Reference Library ≠ Standards Library: Reference Library = patterns and external reference architectures; Standards Library = approved products (constrains Phase D).

04

Governance & Capability Framework Recap

Governance levels

Corporate Governance → IT/Technology Governance → Architecture Governance. Each level nested within the next. Architecture governance is concerned with architecture coherence; IT governance with investment and delivery.

Architecture Board — 3 key functions
1. Review and approve architecture deliverables · 2. Grant dispensations for non-conforming implementations · 3. Maintain the Governance Log. Formal authority comes from signed Architecture Contracts.
Architecture Contracts — key facts
Drafted in Phase F · Signed in Phase G · Two types: Design Contracts and Business/IT Interface Contracts · Without them, governance is advisory only.
Capability Framework — used in
The Preliminary Phase establishes the architecture capability. Covers: Organisation, Processes, Skills, Principles. Maturity: 0 (None) → 3 (Defined, minimum effective) → 5 (Optimising).
Architecture Centre of Excellence
Dedicated unit housing the EA function. Develops capability, provides services, supports governance. Placement close to executive leadership is essential for authority.
⭐ Exam tip

Conformance = matches the architecture. Compliance = meets all governance requirements. A conformant project can still be non-compliant. Dispensation = formal permission to deviate from a conformance requirement — must be approved by Architecture Board and logged in Governance Log.

05

Common Exam Traps & Tricky Distinctions

These are the distinctions that catch candidates out most often. Study these carefully before Day 20.

❌ "The Architecture Roadmap is first produced in Phase A."
WRONG. The Architecture Roadmap is first produced in Phase E. Phase A produces the Architecture Vision and Statement of Architecture Work.
❌ "Requirements Management has a defined start and end like other phases."
WRONG. Requirements Management is the only continuous phase — it runs throughout the entire ADM cycle, never stopping.
❌ "Security architecture is a Phase D concern."
WRONG. Security is a cross-cutting concern addressed in every phase — policies in B, data classification in C, platform services in D, compliance in G.
❌ "SBBs are defined in Phases B, C, and D alongside ABBs."
WRONG. ABBs are defined in Phases B–D. SBBs are first mapped in Phase E — this is the first point where specific products appear.
❌ "Conformance and compliance mean the same thing."
WRONG. Conformance = matches the architecture. Compliance = meets all governance requirements. A conformant project can still be non-compliant if the architecture missed a regulatory requirement.
❌ "The Statement of Architecture Work is an input to Phase A."
WRONG. The Statement of Architecture Work is the primary output of Phase A, not an input. The input that triggers Phase A is the Request for Architecture Work.
❌ "The Architecture Capability Framework covers how to do architecture design."
WRONG. The Capability Framework covers how to establish the capability to do architecture — organisation, processes, skills, principles. Design is done using the ADM.
❌ "The Reference Library and Standards Library contain the same content."
WRONG. Reference Library = patterns and external reference architectures. Standards Library = approved technology products that constrain Phase D choices.
⭐ Final exam tip

Watch for the word "primary" — "what is the primary output of Phase X?" means the most important single output, not every output. Also watch for "only" — "Requirements Management is the only continuous phase" is a favourite question. And remember: architecture governance operates within IT governance — they are complementary, not competing.
