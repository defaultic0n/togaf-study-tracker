01

Phase C Overview — Two Sub-Domains

Objective

Phase C develops the Information Systems Architecture, which describes how data and applications support the Business Architecture developed in Phase B. It covers two sub-domains — Data Architecture and Application Architecture — as a single combined phase.

Phase C is the second of the three detailed architecture phases (B → C → D). After defining the business in Phase B, Phase C answers the question: what information systems are needed to support that business? This means designing the data assets the business needs and the applications that create, manage, and use that data.

Data Architecture
  • Data entities, their structure and relationships
  • Logical and physical data components
  • Data quality, ownership, governance
  • How data flows across the enterprise
  • Data lifecycle management
Application Architecture
  • Logical and physical application components
  • How applications interact with each other
  • Which applications support which business functions
  • Application integration patterns
  • Application portfolio rationalisation
Why two sub-domains in one phase?

Data and Application are closely coupled — you cannot sensibly design applications without knowing what data they manage, and you cannot design data architecture without knowing which applications will consume it. TOGAF keeps them together to ensure they are designed in tandem, not in isolation.

⭐ Exam tip

Phase C covers both Data and Application architecture — not just one of them. The exam sometimes presents Phase C as covering only Application architecture. That is incorrect. Both sub-domains are addressed in Phase C.

02

Data Architecture

Definition

Data Architecture defines the structure of an organisation's logical and physical data assets and the data management resources. It answers: what data exists, how it is structured, where it lives, who owns it, and how it flows through the enterprise.

Data Architecture is often the most overlooked of the four BDAT domains — yet it is foundational. Applications are built to manage data. Technology infrastructure exists to store and transmit data. Getting the data architecture right is what makes the other three domains coherent.

Data Entity

A thing the enterprise needs to store information about. Examples: Customer, Product, Order, Employee, Invoice. Entities are defined at a logical level — independent of how they are stored physically.

Logical Data Component

A grouping of data entities that belong together conceptually. E.g. "Customer Profile" is a logical data component containing Customer, Address, Contact, and Preference entities.

Physical Data Component

The actual implementation of a logical data component — a specific database, data store, or data service. Physical components are technology-specific; logical components are not.

Data Owner

The business role responsible for a data entity or component — accountable for its accuracy, quality, and appropriate use. Defined in the Business Architecture (Phase B) but operationalised here.

Data Lifecycle

The stages data passes through from creation to archival or deletion — Creation, Storage, Use, Sharing, Archival, Deletion. Data Architecture defines policies for each stage.

Data Architecture artefacts (Phase C — Data)

Key artefacts include: Data Entity / Data Component Catalogue (list of all data entities and components), Data Entity / Business Function Matrix (which business functions create, read, update, delete which data entities — a CRUD matrix), and the Data Dissemination Diagram (how data flows between applications and organisations).

⭐ Exam tip

Know the distinction between logical (technology-independent, conceptual) and physical (technology-specific, implementation) for both data and application components. This logical/physical distinction runs through Phases C and D and is frequently tested.

03

Application Architecture

Definition

Application Architecture defines the individual logical and physical application components and their interactions. It describes how applications are structured, how they interact with each other, and how they support the business functions defined in Phase B.

Application Architecture is not about selecting specific software products — that is a procurement decision. It is about defining the logical application landscape: what capabilities applications must provide, how they should be grouped, and how they should communicate.

Logical Application Component

A self-contained encapsulation of application functionality — defined at a conceptual level, independent of technology choices. E.g. "Customer Relationship Management", "Order Management System", "Reporting Engine".

Physical Application Component

A specific technology implementation of a logical component. E.g. "Salesforce" implements the CRM logical component. Physical components are technology-specific.

Application Platform

The shared technical foundation that application components run on — including middleware, integration services, authentication, and messaging infrastructure.

Application Interface

The defined point of interaction between application components — APIs, message queues, data feeds, file transfers. Interfaces are critical for integration architecture.

Application portfolio rationalisation

Phase C is also where application portfolio rationalisation happens — identifying duplicate applications, legacy applications that should be retired, gaps where no application exists, and consolidation opportunities. This is a key input to Phase E (Opportunities & Solutions).

Application Architecture artefacts

Key artefacts include: Application Portfolio Catalogue (list of all applications), Application / Function Matrix (which applications support which business functions), Application Interaction Diagram (how applications communicate), and Application Communication Diagram (the flows of information between applications).

⭐ Exam tip

Application Architecture defines logical components first — then physical. Don't select specific products in Phase C; that comes later. The logical-then-physical approach applies to both data and application architecture.

04

Application Interaction Matrix & Data Dissemination

Two artefacts from Phase C are particularly important for the exam: the Application Interaction Matrix and the Data Dissemination Diagram. Both capture how information moves across the enterprise.

Application Interaction Matrix

A matrix that maps applications against each other to show which applications interact, the nature of those interactions (send data, receive data, both), and the data exchanged. It is the application-level equivalent of the Business Interaction Matrix from Phase B.

Application Interaction MatrixMatrixWhich applications communicate and what data they exchangeIdentifying integration complexity and redundancy

Data Dissemination Diagram

This diagram traces how a specific data entity flows from its authoritative source (the system that creates and owns it) through the various applications and systems that consume it. It makes data lineage visible — essential for data governance, compliance, and designing data integration.

How Phase C artefacts connect to Phase B

The Application / Function Matrix maps applications (Phase C) directly back to business functions (Phase B). The Data Entity / Business Function Matrix shows which Phase B business functions create or consume which Phase C data entities. These linkages ensure the information systems architecture stays grounded in the business architecture it is meant to support.

⭐ Exam tip

The Application Interaction Matrix and Data Dissemination Diagram are the two most tested Phase C artefacts. Know what each shows: the matrix maps application-to-application interactions; the diagram traces data flows from source to consumer.

05

Phase C Inputs, Steps & Outputs

Like all ADM phases, Phase C follows the baseline → target → gap analysis pattern. It does this twice — once for Data Architecture and once for Application Architecture — before producing its outputs.

Phase C steps

1. Select reference models, viewpoints, and tools. 2. Develop Baseline Data Architecture. 3. Develop Target Data Architecture. 4. Perform Data Architecture Gap Analysis. 5. Develop Baseline Application Architecture. 6. Develop Target Application Architecture. 7. Perform Application Architecture Gap Analysis. 8. Define candidate Roadmap components. 9. Resolve impacts across the Architecture Landscape. 10. Conduct formal stakeholder review. 11. Finalise the Information Systems Architecture.

Key Inputs
  • Architecture Vision (Phase A)
  • Business Architecture (Phase B) ★
  • Business Interaction Matrix (Phase B)
  • Architecture Principles
  • Architecture Repository
  • Relevant industry data models
  • Existing IS documentation
Key Outputs
  • Baseline IS Architecture (Data + App) ★
  • Target IS Architecture (Data + App) ★
  • IS Architecture Gap Analysis ★
  • Application Interaction Matrix
  • Data Dissemination Diagram
  • Architecture Definition Document (C chapter)
  • Updated Architecture Repository
Phase B feeds Phase C

The Business Architecture from Phase B is the primary input to Phase C. Specifically: the Business Interaction Matrix identifies which business units interact — this directly shapes which applications need to integrate. The Actor / Role Matrix identifies who needs to access what — shaping application access and authorisation design.

⭐ Exam tip

Phase C follows the same baseline → target → gap pattern as Phase B — but does it twice: once for Data, once for Application. The gaps feed Phase E alongside the Phase B gaps. The Business Architecture (Phase B) is the critical input that scopes all Phase C work.
