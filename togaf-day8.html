01

Phase F Objective — From Outline to Definitive Plan

Objective

Phase F finalises a detailed, prioritised, and costed Implementation and Migration Plan — the authoritative delivery document that governs how the enterprise transitions from its Baseline Architecture to its Target Architecture.

Phase E created the outline — work packages, transition architectures, an initial roadmap. Phase F makes it definitive. The key word is prioritised: not every work package can be started at once. Phase F establishes the order of implementation based on dependencies, business value, cost, and risk.

Phase E produces
  • Initial Architecture Roadmap
  • Work Package definitions
  • Transition Architecture definitions
  • Initial Implementation plan
  • Project Context Diagram
Phase F refines into
  • Finalised Architecture Roadmap
  • Prioritised work packages (sequenced)
  • Confirmed Transition Architectures
  • Definitive Migration Plan
  • Approved Implementation schedule
Why Phase F is distinct from Phase E

Phase E asks: "What work needs to be done and in what order?" Phase F asks: "Exactly when, by whom, at what cost, with what dependencies, and with what governance?" Phase F goes significantly deeper into the practical planning required to mobilise delivery.

⭐ Exam tip

Phase E = identifies and defines the delivery plan. Phase F = prioritises, details, and finalises it. The Implementation and Migration Plan is finalised in Phase F, not Phase E. This distinction is tested frequently.

02

Assessing Dependencies, Costs & Benefits

A core activity of Phase F is the rigorous assessment of each work package — understanding its dependencies, estimating its cost, and quantifying the business benefits it delivers. Without this analysis, the implementation plan cannot be properly prioritised.

Dependency assessment

Not all work packages are independent. Some projects cannot begin until others are complete. Phase F maps all dependencies — both hard dependencies (must-have sequencing) and soft dependencies (preferred sequencing) — to ensure the implementation plan is executable.

Technical Dependencies

One work package depends on a technical output from another. E.g. a new application cannot be deployed until the infrastructure upgrade it needs is complete.

Business Dependencies

A business capability change cannot be implemented until organisational changes are in place. E.g. a new service model cannot be launched until staff are trained.

Data Dependencies

An application migration depends on data migration completing first. Data quality issues in the baseline may delay dependent work packages.

Resource Dependencies

Two work packages compete for the same scarce resource — a specialist team, a key system, or a budget cycle. Resource conflicts must be resolved in the migration plan.

Cost–benefit analysis

Every work package is assessed for cost (implementation, operations, transition) and business benefit (revenue, savings, risk reduction, compliance). This analysis produces a cost-benefit ranking that, combined with dependency constraints, determines the optimal sequencing of work.

Risk assessment per work package

Phase F also performs a detailed risk assessment for each work package — identifying technical risks, business disruption risks, and programme-level risks. Risk mitigation strategies are defined and factored into the implementation schedule.

⭐ Exam tip

The prioritisation of work packages in Phase F is based on a combination of: dependency constraints (what must come first), business value (what delivers most benefit), cost (what is affordable), and risk (what is safest to implement when). All four factors play a role.

03

Migration Planning Techniques

Phase F uses a set of established planning techniques to turn the outline from Phase E into a detailed, executable plan. These techniques bring analytical rigour to what could otherwise be an ad hoc planning exercise.

Key techniques

TOGAF identifies several techniques commonly used in Phase F migration planning — from financial modelling to business value assessment to capability gap scoring.

TechniqueWhat it doesUsed for
Business Value AssessmentScores each work package against business benefit criteriaPrioritisation ranking
Dependency ModellingMaps hard and soft dependencies between work packagesSequencing constraints
Cost–Benefit AnalysisEstimates NPV, ROI, payback period for each packageFinancial justification
Risk Assessment MatrixScores implementation risk vs business impactRisk-based scheduling
Capability Gap ScoringRanks work packages by how much capability gap they closeStrategic alignment
Portfolio OptimisationBalances the mix of work in each time horizonResource and budget levelling
Prioritisation matrix

A common Phase F output is a prioritisation matrix — plotting each work package against two dimensions (typically business value and implementation effort or risk). High-value, low-risk work packages are scheduled earliest. High-value, high-risk packages are planned with additional mitigation. Low-value packages may be descoped or deferred.

⭐ Exam tip

Know that Phase F uses multiple planning techniques — it is not just a sequencing exercise. Business value assessment, cost-benefit analysis, and risk assessment all feed into the final prioritised plan. The exam sometimes asks which phase performs detailed cost-benefit analysis — the answer is Phase F.

04

Architecture Roadmap Refinement

The Architecture Roadmap, first created in Phase E, is refined and finalised in Phase F. The refinement process incorporates the dependency analysis, cost-benefit rankings, and risk assessments to produce a time-sequenced, resource-aware plan.

From initial to finalised roadmap

The Phase E roadmap shows what needs to happen and in approximately what order. The Phase F roadmap shows exactly when each work package will be executed, by whom, at what cost, and under what governance.

1
Confirm Transition Architecture states
Review the transition architectures from Phase E against the detailed dependency and cost analysis. Adjust if needed.
2
Sequence work packages
Apply dependency constraints and prioritisation rankings to produce a sequenced delivery schedule.
3
Assign to time horizons
Group work packages into near-term (0–12 months), mid-term (12–24 months), and long-term (24+ months) horizons.
4
Resource and budget plan
Assign resources to each work package. Level the resource plan to avoid overloads. Confirm budget envelope for each time horizon.
5
Finalise the Roadmap
Produce the definitive Architecture Roadmap — a time-phased, resource-aware, costed, and governed plan ready for stakeholder approval.
⭐ Exam tip

The Architecture Roadmap is created in Phase E and finalised in Phase F. If asked "which phase finalises the roadmap?" — Phase F. If asked "which phase first creates it?" — Phase E. Both are tested.

05

Stakeholder Reviews, Sign-off & Phase F Outputs

Phase F concludes with formal stakeholder reviews and governance sign-off. The Implementation and Migration Plan — now detailed and costed — must be reviewed by the Architecture Board and approved by the sponsoring organisation before delivery can begin.

The sign-off process

Phase F's sign-off ensures that the delivery plan has formal organisational endorsement. It also marks the transition from architecture governance to implementation governance — the Architecture Board hands over to programme governance structures for Phase G onward.

Architecture Board review

The Architecture Board reviews the finalised Migration Plan and Roadmap against the Architecture Vision and principles. Any deviations require formal dispensation.

Sponsor and executive sign-off

The sponsoring executive and key business stakeholders formally approve the plan. This commits organisational resources — budget, staff, time — to the delivery programme.

Transition to Phase G

Once the plan is approved, Phase G — Implementation Governance — takes over. Phase G monitors delivery against the approved Architecture Plan throughout implementation.
