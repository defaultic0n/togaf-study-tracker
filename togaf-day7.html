01

Phase E Objective — Design to Delivery

Objective

Phase E identifies the preferred combination of building blocks — projects, programmes, transition architectures, and work packages — to deliver the Target Architecture defined in Phases B, C, and D. It is the first phase focused on delivery planning rather than architecture design.

Phase E represents a fundamental shift in the ADM. Phases B, C, and D asked: what should the architecture look like? Phase E asks: how do we get there? It takes the abstract gap analyses from three detailed phases and turns them into a concrete, actionable roadmap.

Why Phase E is a pivot

Up to Phase D, the architecture team has been working primarily in the realm of design — documenting baseline states, defining target states, and identifying gaps. Phase E is where architects hand over to delivery organisations. The work packages and transition architectures defined here become inputs to real projects and programmes.

Architecture Roadmap

A time-sequenced view of the work packages and projects needed to move from the Baseline Architecture to the Target Architecture. First produced in Phase E; refined and detailed in Phase F.

Implementation and Migration Plan

A detailed, prioritised plan showing the sequence of projects, their dependencies, costs, benefits, and timelines. Phase E produces an initial version; Phase F refines it into the definitive plan.

⭐ Exam tip

Phase E is the first of the delivery planning phases (E and F). It is not an architecture design phase. The Architecture Roadmap is first created in Phase E. If an exam question asks "when is the Architecture Roadmap first produced?" — the answer is Phase E.

02

Transition Architectures

One of the most important concepts introduced in Phase E is the Transition Architecture. When the gap between the Baseline Architecture and the Target Architecture is too large to close in a single step, transition architectures define the intermediate states the enterprise passes through.

Definition

A Transition Architecture is a formally defined intermediate architecture state — a coherent, stable configuration of the enterprise at a point in time between the baseline and target. Each transition architecture represents a valuable and achievable milestone in the journey to the target.

Baseline Architecture
Current state
→
Transition Architecture 1
e.g. 12 months
→
Transition Architecture 2
e.g. 24 months
→
Target Architecture
Future state

Transition architectures are important because they:

Reduce risk

Instead of one massive, high-risk transformation, the organisation makes a series of smaller, more manageable changes. Each transition architecture is independently achievable and delivers real business value.

Enable incremental value delivery

Each transition architecture delivers real capability improvements to the business even if the full target architecture is years away. Stakeholders see progress and value is realised throughout the journey.

Allow course correction

Between each transition architecture, the organisation can pause, review, and adjust the plan. If business priorities change, the transition plan can be modified without abandoning all work done so far.

Manage operational continuity

The business continues to operate during the transformation. Each transition architecture is a coherent, stable state — not a half-finished mess that disrupts operations.

⭐ Exam tip

A Transition Architecture is not the same as a target architecture — it is an intermediate stable state between baseline and target. They are defined in Phase E and sequenced into the Architecture Roadmap. When a question mentions "intermediate architecture states," the answer is Transition Architectures.

03

Work Packages & Implementation Projects

Phase E organises the delivery work into Work Packages — discrete, manageable units of implementation work that collectively close the gaps identified in Phases B, C, and D.

Work Package definition

A Work Package is a group of related activities and deliverables required to implement a portion of the architecture. Each work package defines what needs to be done, what will be delivered, what it depends on, and how long it will take.

📦 Work Package — Typical Contents
  • Name and description of the work
  • Architecture building blocks to be implemented
  • Business value delivered when complete
  • Dependencies on other work packages
  • Estimated effort, cost, and duration
  • Risk assessment for this work package
  • Stakeholders affected by this work
  • Acceptance criteria — how we know it's done
Work Packages → Projects → Programmes

Work packages are grouped into implementation projects — formal project management structures with assigned resources, budgets, and governance. Multiple related projects may be grouped into a programme. These projects and programmes are then sequenced onto the Architecture Roadmap in Phase E and detailed in Phase F.

Project Context Diagram

A key Phase E artefact — the Project Context Diagram shows the dependencies between implementation projects and their relationships to the transition architectures. It makes the delivery sequencing visible and is used for portfolio planning and stakeholder communications.

⭐ Exam tip

Work packages are defined in Phase E and refined/prioritised in Phase F. The Project Context Diagram (Phase E artefact) shows project dependencies and relationships to transition architectures. Know the distinction between Phase E (defines the plan) and Phase F (prioritises and details it).

04

Consolidated Gap Analysis

Phase E performs a Consolidated Gap Analysis — bringing together all the individual gap analyses from Phases B, C, and D into a single comprehensive view. This consolidation is essential because gaps across architecture domains interact and overlap in ways that individual domain analyses cannot capture.

Why consolidation matters

A gap in Business Architecture (Phase B) may be addressable only after a gap in Application Architecture (Phase C) is closed. A technology upgrade (Phase D) may unlock multiple application and business capability improvements. The consolidated view reveals these dependencies — enabling rational sequencing of work.

Gap Analysis categories revisited

In Phase E, gaps are revisited from all three domains. Each gap is evaluated against: What work package closes it? · Which transition architecture does it belong to? · What are the dependencies with other gaps? · What is the business priority? · What is the estimated cost and risk?

The output of consolidated gap analysis is a gap-to-project mapping — every gap from Phases B, C, and D is assigned to a work package, and every work package is assigned to a project. This mapping ensures no gap is forgotten and no project exists without traceable architectural justification.

Architecture Building Blocks vs Solution Building Blocks

Phase E is also where TOGAF's distinction between Architecture Building Blocks (ABBs) and Solution Building Blocks (SBBs) becomes operationally important. ABBs are technology-independent architecture components (defined in Phases B–D). SBBs are specific products and technologies that implement ABBs. Phase E maps ABBs to candidate SBBs — the first point at which specific implementation solutions are identified.

⭐ Exam tip

The Consolidated Gap Analysis in Phase E brings together gaps from all three of Phases B, C, and D. This is the first time all gaps are seen together. Phase E is also where Architecture Building Blocks (ABBs) are mapped to Solution Building Blocks (SBBs) — technology-independent to technology-specific.

05

Solution Selection Criteria & Phase E Outputs

Phase E doesn't just identify what work needs to be done — it also establishes the criteria by which solutions are evaluated and selected. This brings rigour to solution selection and prevents purely technical or politically-driven choices.

Fitness for Purpose
Does the solution actually deliver the required capability and close the identified gap?
Alignment to Principles
Does the solution comply with the Architecture Principles defined in the Preliminary Phase?
Cost and Value
Is the cost proportionate to the business value delivered? What is the total cost of ownership?
Risk and Feasibility
Is the solution technically and organisationally feasible? What risks does it introduce?
Vendor and Market
Is the solution from a viable vendor? Is it supported long-term? Does it align to technology standards?
Reuse Potential
Can this solution serve multiple work packages or business areas? Maximising reuse reduces cost.
Key Inputs
  • Architecture Definition Document (B+C+D) ★
  • Gap Analysis results (B, C, D) ★
  • Architecture Vision
  • Architecture Principles
  • Architecture Repository
  • Enterprise Continuum (SBBs)
Key Outputs
  • Architecture Roadmap (initial) ★
  • Work Package definitions ★
  • Transition Architecture definitions ★
  • Implementation and Migration Plan (initial)
  • Project Context Diagram
  • Capability Assessment
⭐ Exam tip

Phase E produces the initial Architecture Roadmap — first appearance in the ADM. It also defines Transition Architectures and Work Packages. Phase F then refines and prioritises all of these. Know what is created in E versus what is refined in F.
