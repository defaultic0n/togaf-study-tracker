01

Phase G Objective & Role

Objective

Phase G ensures that the implementation projects deliver architecture conformance — that what gets built matches the Target Architecture defined in Phases B through D and planned in Phases E and F. It provides ongoing architecture oversight throughout the delivery programme.

Phase G is a significant change of mode for the architecture team. In Phases B through F, architects were designers and planners. In Phase G, they become governance authorities — reviewing what the delivery teams are building, checking it against the architecture contracts, and raising issues when implementations diverge from the approved architecture.

What Phase G is not

Phase G is not project management — that's the delivery programme's responsibility. Phase G is not quality assurance in the technical testing sense. Phase G is specifically architecture conformance oversight — the Architecture Board and architecture team ensuring that implementations stay true to the approved architectural design.

Architecture Oversight

Monitoring ongoing implementation projects to ensure they conform to the approved architecture. Raises architecture-level concerns, not project management issues.

Governance Body

The Architecture Board plays the central role in Phase G — reviewing implementation decisions, granting dispensations, and escalating non-conformances.

Architecture Contract Enforcement

The Architecture Contracts drafted in Phase F are now actively enforced — delivery teams must demonstrate compliance with their contract commitments.

⭐ Exam tip

Phase G is the first governance phase in the ADM — it covers the delivery period, not the design period. It uses Architecture Contracts (drafted in Phase F) as the baseline for compliance checks. Phase H (tomorrow) handles changes after delivery.

02

Architecture Contracts

Definition

An Architecture Contract is a joint agreement between development partners and sponsors on the deliverables, quality, and fitness-for-purpose of an architecture. It defines the specific architecture requirements that an implementation project must satisfy.

Architecture Contracts are the foundational tool of Phase G. They are drafted in Phase F and activated in Phase G. Think of them as the formal handshake between the architecture team and the delivery team — the architecture team says "here is what must be built," and the delivery team agrees to build it to that specification.

📋 Architecture Contract — Typical Contents
  • Scope of the work to be performed
  • Architecture requirements the deliverable must meet
  • Standards and guidelines that must be followed
  • Performance criteria and quality measures
  • Acceptance criteria — how conformance will be verified
  • Governance and reporting arrangements
  • Escalation procedures for non-conformances
  • Any approved dispensations from architecture requirements
Types of Architecture Contracts

TOGAF identifies two main contract types. Development Partner contracts govern third-party suppliers or internal teams implementing architecture components. Business User contracts govern the business units that will consume and operate the implemented architecture. Both types define specific conformance obligations.

Contracts and governance

Architecture Contracts are reviewed and monitored by the Architecture Board. When a delivery team cannot meet a contract requirement — due to technical constraints, cost, or changed circumstances — they submit a request for dispensation (a formal exception to a contract requirement). The Architecture Board decides whether to grant it.

⭐ Exam tip

Architecture Contracts are drafted in Phase F and enforced in Phase G. They define what must be built to conform to the architecture. Non-conformances are escalated to the Architecture Board. A dispensation is a formal exception granted when full conformance cannot be achieved.

03

Conformance vs Compliance Reviews

Two terms that look similar but mean very different things in Phase G: conformance and compliance. Getting these right matters for the exam.

TermDefinitionAssessed againstWho performs it
Architecture ConformanceThe degree to which an implementation matches the approved architecture designs and contractsArchitecture Definition Document, Architecture ContractsArchitecture team / Architecture Board
Architecture ComplianceAdherence to external standards, regulations, legal requirements, and organisational policiesExternal standards bodies, regulations, internal governance policiesCompliance function, legal, audit
Why the distinction matters

An implementation can be architecturally conformant (built exactly as designed) but still not compliant (it violates GDPR, for example). Conversely, an implementation can be compliant with regulations but non-conformant with the approved architecture. Phase G checks conformance; compliance is the organisation's legal and audit function.

Levels of conformance

TOGAF recognises that not all implementations will achieve 100% architectural conformance. The Architecture Board may accept various levels: Fully conformant (meets all requirements), Partially conformant (meets most, with approved exceptions), Non-conformant with dispensation (formally approved departure from the architecture), Non-conformant without dispensation (a governance issue requiring escalation).

⭐ Exam tip

Conformance = matching the architecture design. Compliance = meeting external standards and regulations. Phase G checks conformance. Compliance is separate. A very common exam distractor conflates these two concepts.

04

Architecture Compliance Review Process

The Architecture Compliance Review is the primary mechanism Phase G uses to check that implementations are on track. It is a structured review conducted at key milestones in the implementation programme.

Definition

An Architecture Compliance Review is a scrutiny of the compliance of a specific project against established architectural criteria — assessing conformance to the Architecture Contract and the Architecture Definition Document.

1
Schedule the review
Identify the project milestone at which the review will occur — typically at key delivery checkpoints defined in the Architecture Contract.
2
Prepare review materials
Delivery team provides architecture-relevant documentation — design specs, test evidence, deviation logs. Architecture team prepares the compliance checklist.
3
Conduct the review
Architecture team assesses the implementation against the Architecture Contract and Architecture Definition Document. Technical interviews may supplement documentary review.
4
Document findings
Produce a Compliance Assessment report — rating each requirement as conformant, partially conformant, non-conformant, or not applicable. Note approved dispensations.
5
Escalate and resolve
Non-conformances are escalated to the Architecture Board. The Board decides: require remediation, grant dispensation, or escalate to sponsor for a decision on whether to accept the non-conformance.
Compliance Assessment report

The key output of a compliance review is the Compliance Assessment — a formal document recording the review findings, conformance ratings, approved dispensations, and required remediation actions. This document is stored in the Architecture Repository and feeds Phase H if the non-conformance triggers an architecture change.

⭐ Exam tip

The Architecture Compliance Review is the primary Phase G activity. Its output is the Compliance Assessment. Non-conformances escalate to the Architecture Board. If a non-conformance is significant enough to require an architecture change, it triggers Phase H.

05

Phase G Inputs, Steps & Outputs

Phase G runs throughout the entire delivery programme — not as a one-time activity, but as an ongoing governance overlay on every implementation project.

Key Inputs
  • Implementation and Migration Plan (Phase F) ★
  • Architecture Contracts (drafted Phase F) ★
  • Architecture Definition Document
  • Architecture Repository
  • Architecture Vision and Principles
  • Governance framework
Key Outputs
  • Architecture Contracts (signed and active) ★
  • Compliance Assessments ★
  • Architecture Repository (updated)
  • Change Requests (for Phase H)
  • Statement of Architecture Work (closed)
  • Architecture-compliant implementations
Phase G → Phase H connection

When a compliance review identifies a significant non-conformance — or when a delivery team proposes a change that would deviate from the architecture — Phase G raises a Change Request. This Change Request feeds directly into Phase H — Architecture Change Management, which decides whether to update the architecture, grant a dispensation, or require the implementation to be redone.

⭐ Exam tip

Phase G produces Change Requests that trigger Phase H. Phase G checks conformance during delivery; Phase H handles changes to the architecture. These two governance phases work as a pair — Phase G monitors, Phase H responds to deviations.
