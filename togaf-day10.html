01

Phase H Objective & Context

Objective

Phase H establishes and manages an Architecture Change Management process to ensure the enterprise architecture lifecycle is maintained in response to new developments, ensuring the architecture remains current, relevant, and fit for purpose.

After Phase G has overseen delivery, the enterprise enters an operational steady state with the new architecture in place. But change is constant — business strategies evolve, new technologies emerge, regulations change, and implementation discovers gaps the architecture didn't anticipate. Phase H is the mechanism that keeps the architecture alive and responsive to all of these changes.

Phase H's position in the ADM

Phase H is the last lettered phase in the ADM cycle. But it is not an end — it is the gateway back to the beginning. When Phase H determines that a change is significant enough to require a new architecture cycle, it generates a new Request for Architecture Work — triggering a new ADM cycle starting from Phase A.

Architecture Maintenance

Minor updates to the architecture that don't require re-running the full ADM cycle — small adjustments, clarifications, additions of new standards or guidelines.

Architecture Redesign

Significant changes that require a new or partial ADM cycle — major business transformation, fundamental technology shifts, or significant scope changes.

⭐ Exam tip

Phase H is the last ADM phase and loops back to the beginning. When significant changes occur, Phase H generates a new Request for Architecture Work — triggering a new ADM cycle. This makes the ADM a continuous cycle, not a linear process.

02

Change Management Triggers & Types

Phase H is activated by change triggers — events or circumstances that suggest the current architecture may need to be updated. TOGAF categorises these into business-driven and technology-driven triggers.

📈
Business strategy change
Mergers, acquisitions, new market entry, strategic pivots, or major changes to business model that invalidate architecture assumptions.
📋
New regulations or compliance requirements
New laws, standards, or regulatory requirements that impose new constraints or obligations on the architecture.
💻
Technology change
Emergence of disruptive new technologies, product end-of-life, platform migrations, or major version upgrades that affect the architecture.
🔍
Implementation discoveries
Findings during Phase G that reveal architecture gaps, incorrect assumptions, or unforeseen constraints that require architecture changes.
⚡
Lessons learned
Completed projects reveal improvements to the architecture — better patterns, standards updates, or reusable building blocks identified.
🌍
External market developments
Competitive landscape changes, new market standards, partner ecosystem changes, or industry disruption requiring strategic architecture response.
Two change categories

All change requests fall into two categories. Business-driven changes originate from strategy shifts, regulatory requirements, or market conditions. Technology-driven changes originate from infrastructure changes, product lifecycles, security vulnerabilities, or platform evolution. Each category may require different governance handling.

⭐ Exam tip

Know the difference between business-driven and technology-driven change triggers. Both types flow through the Architecture Change Request process in Phase H — but they may have different urgency, impact scope, and required governance response.

03

Dispensations & Exceptions

Not all deviations from the architecture require a full change cycle. Some can be handled through the formal dispensation and exception processes — controlled ways to permit specific departures from the architecture without reopening the full design.

Dispensation

A dispensation is a formal, time-limited permission from the Architecture Board to deviate from an architecture requirement. It is not a permanent change to the architecture — it is a controlled exception with defined boundaries and an expiry.

Why dispensations are needed

Implementation constraints may make full conformance impractical — cost, technical feasibility, timelines. A dispensation allows work to proceed while maintaining governance visibility over the exception.

Dispensation vs permanent change

A dispensation is temporary — it expires. If the exception becomes permanent or widespread, it should become a formal architecture change instead. The Architecture Board monitors dispensations to identify patterns.

Dispensation record

Every dispensation is recorded in the Architecture Repository's Governance Log — who requested it, what was permitted, the rationale, expiry date, and conditions. This creates an auditable exception record.

Exception escalation

If a dispensation is refused or if the non-conformance cannot be accommodated by any dispensation, it escalates to a full Architecture Change Request — potentially triggering a new ADM cycle.

Dispensations and the Governance Log

The Governance Log in the Architecture Repository records all dispensations, exceptions, and compliance decisions. It provides the audit trail that demonstrates the architecture governance process is working — essential for regulated industries where proof of governance is required.

⭐ Exam tip

A dispensation is a temporary, formal, board-approved exception — not a permanent architecture change. Dispensations are recorded in the Governance Log. If a dispensation pattern becomes persistent, it should be converted into a formal Architecture Change Request.

04

Architecture Change Request Process

The Architecture Change Request (ACR) is the formal mechanism for proposing and managing changes to the architecture. It ensures all changes are assessed, governed, and traceable.

📥 Change Request received — from Phase G, business, or technology trigger
↓
🔍 Impact Assessment — scope, cost, risk, affected components
↓
⚖️ Architecture Board Review — assess significance and recommend response
↓
📋 Decision — Maintenance update, Redesign cycle, Dispensation, or Rejection
↓
🔄 If Redesign → new Request for Architecture Work → new ADM cycle from Phase A
Technology impact assessment

For technology-driven changes, the impact assessment must trace the change through all affected architecture domains — from Technology (Phase D) up through Application (Phase C), Data (Phase C), and Business (Phase B). A new technology standard, for example, may affect application interfaces, data schemas, and ultimately business processes.

Business impact assessment

For business-driven changes, the impact assessment works top-down through the architecture — from Business (Phase B) through Information Systems (Phase C) and Technology (Phase D). A business strategy change triggers ripple effects across the entire BDAT stack.

⭐ Exam tip

The key output of Phase H for significant changes is a new Request for Architecture Work — the same document that triggers Phase A. This is how the ADM cycle restarts. Phase H is what makes architecture management continuous rather than one-time.

05

Maintenance vs Redesign & Phase H Outputs

The most critical decision in Phase H is whether a change requires architecture maintenance — a minor update handled within Phase H — or architecture redesign — a significant change requiring a new ADM cycle.

Architecture Maintenance
  • Incremental, non-breaking changes
  • Updates to standards or guidelines
  • Adding new building blocks to existing patterns
  • Minor scope extensions within current architecture
  • Technology version upgrades maintaining compatibility
  • Clarifications and corrections to architecture documentation
Architecture Redesign
  • Major business transformation or merger
  • Fundamental change to architecture principles
  • Disruptive technology requiring architecture rethink
  • Significant change to scope of the architecture
  • Multiple components requiring coordinated redesign
  • Changes that invalidate existing transition architectures
Key Inputs to Phase H
  • Change Requests (from Phase G) ★
  • Architecture Repository (Governance Log)
  • Architecture Definition Document
  • Architecture Vision
  • Business and technology change drivers
Key Outputs of Phase H
  • Architecture updates (maintenance) ★
  • New Request for Architecture Work (redesign) ★
  • Dispensations (for approved exceptions)
  • Governance Log updates
  • Updated Architecture Repository
The ADM as a continuous cycle

Phase H completes the ADM cycle — but more importantly, it restarts it. The Preliminary Phase sets up the architecture capability. Phase A through F designs and plans the architecture. Phase G governs delivery. Phase H responds to change. A new Request for Architecture Work from Phase H loops back into Phase A — making the ADM perpetually relevant as the enterprise evolves.

⭐ Exam tip

Phase H has two responses: Maintenance (minor update, no new ADM cycle) and Redesign (significant change, new ADM cycle via Request for Architecture Work). The decision criteria is the scope and impact of the change. This distinction is tested in scenario-based exam questions.
