01

Technical Reference Model (TRM) Overview

What the TRM is

The Technical Reference Model (TRM) is TOGAF's Foundation Architecture for technology. It provides a model and taxonomy of generic platform services that any enterprise can use as the basis for its own technology architecture. It lives at the Foundation level of the Enterprise Continuum — the most generic, applicable-to-all-organisations position.

The TRM was originally developed by The Open Group to provide a vendor-neutral framework for technology architecture. It defines the types of technology services an enterprise needs — without prescribing specific products. This makes it a stable foundation that organisations customise upward toward their own organisation-specific architecture.

Application Software
Business applications that use platform services to deliver business capabilities. Examples: ERP, CRM, HR systems. This layer represents the actual business functionality delivered through technology.
Application Platform
The collection of technology services that application software depends on. This is the core of the TRM — a taxonomy of platform service types covering data management, security, network, OS, and more.
Communications Infrastructure
The underlying network and communications infrastructure — LANs, WANs, internet, mobile networks — that connects the Application Platform and Application Software to users and other systems.
TRM and the Enterprise Continuum

The TRM sits at the Foundation Architecture level — the most generic position in the Architecture Continuum. Every organisation can use the TRM as its starting point for technology architecture. Organisations then extend and specialise it toward their own Technology Architecture in Phase D.

⭐ Exam tip

The TRM has two main components: the taxonomy of platform services and the TRM graphic (three-layer diagram). It sits at the Foundation Architecture level of the Enterprise Continuum. It defines service types — not specific products.

02

TRM Taxonomy & Application Platform Services

The Application Platform service taxonomy

The Application Platform layer of the TRM contains a detailed taxonomy of technology service types. These service types represent the categories of capability any application needs from its underlying platform. The exam tests knowledge of these categories.

Data Interchange Services
Document and data interchange, data format conversion, electronic data interchange (EDI).
Data Management Services
Database management, data warehousing, data access, query services, data replication.
Graphics & Imaging Services
2D/3D graphics, image processing, video display, multimedia presentation.
User Interface Services
Window management, display services, dialogue support, user interaction management.
Network Services
Communications transport, network addressing, session management, protocol services.
Operating System Services
Kernel services, memory management, process management, file system, device drivers.
Security Services
Identification and authentication, authorisation, access control, encryption, audit logging.
Location & Directory Services
Directory services (LDAP), naming services, geolocation, resource discovery.
Systems & Network Management
Configuration management, monitoring, performance management, fault management.
Transaction Processing Services
Transaction management, distributed transaction coordination, rollback/commit.
⭐ Exam tip

You don't need to memorise every service. Focus on knowing the categories: Data Management, Data Interchange, Security, Network, OS, Location/Directory, Transaction Processing, User Interface, Graphics/Imaging, Systems Management. Security is explicitly a platform service in the TRM — reinforcing that it's cross-cutting.

03

Integrated Information Infrastructure Reference Model (III-RM)

What the III-RM is

The Integrated Information Infrastructure Reference Model (III-RM) is TOGAF's second Foundation Architecture reference model. Where the TRM addresses platform services for a single enterprise, the III-RM addresses the requirements for information sharing across enterprise boundaries — the infrastructure needed for "boundaryless information flow."

The III-RM was developed to address the reality that modern enterprises don't operate in isolation. They need to share information with partners, suppliers, customers, regulators, and other external organisations. The III-RM provides the reference architecture for this cross-enterprise information infrastructure.

Business Applications (III-RM)

The enterprise's business applications that need to exchange information across boundaries. May span multiple organisations and technology environments.

Infrastructure Applications

The middleware and integration applications that enable information exchange — message brokers, integration platforms, API gateways, identity federation services.

Application Platform (III-RM)

Extended platform services supporting cross-enterprise integration — federated identity management, secure messaging, data translation, cross-domain security services.

Relationship to TRM

The III-RM extends and complements the TRM. The TRM defines platform services for a single enterprise environment; the III-RM extends this to cover the additional services needed for cross-enterprise information sharing. They are used together, not as alternatives.

III-RM in the Enterprise Continuum

Like the TRM, the III-RM sits at the Foundation Architecture level of the Enterprise Continuum. Together, the TRM and III-RM form TOGAF's complete Foundation Architecture reference models — the most generic building blocks available for any enterprise's technology architecture.

⭐ Exam tip

The III-RM addresses cross-enterprise information sharing — it extends the TRM for networked and distributed environments. Both TRM and III-RM sit at the Foundation Architecture level. The III-RM is associated with the concept of "boundaryless information flow."

04

Boundaryless Information Flow

The concept

Boundaryless Information Flow is The Open Group's vision for the ideal state of enterprise information infrastructure: the ability to access integrated information to support business needs, anytime, anywhere, without barriers between systems, organisations, or geographies.

Boundaryless information flow does not mean security-free or open access to everything. It means that the right information can flow to the right people at the right time — with all appropriate security, governance, and compliance controls in place. The barriers removed are technical and organisational impediments, not security controls.

Interoperability

Systems can exchange information seamlessly using agreed standards, protocols, and data formats — regardless of the vendor, platform, or location of each system.

Integration

Information from multiple systems and organisations is aggregated, correlated, and presented in a unified way — providing a coherent view of data that would otherwise be fragmented across silos.

Accessibility

Authorised users can access the information they need from any device, any location, at any time — through appropriate channels and with appropriate authentication.

Security-controlled openness

Information flows are governed by security policies — who can access what, under what conditions. Boundaryless flow is achieved within a governed security framework, not despite it.

⭐ Exam tip

Boundaryless information flow is the guiding vision of the III-RM. It means removing technical and organisational barriers to information sharing — while maintaining security controls. It does not mean unrestricted open access.

05

Using Reference Models in Phase D

Phase D — Technology Architecture — is the primary ADM phase where the TRM and III-RM are applied. Architects use these reference models as the starting point for defining the organisation's technology architecture.

How the TRM is used in Phase D

In Phase D, the TRM provides a checklist and taxonomy. For each application identified in Phase C, the architect maps which Application Platform service types are needed. This ensures no platform services are overlooked and provides a structured framework for technology selection. The Standards Library then constrains which specific products are chosen for each service type.

Step 1: Map application needs to TRM services

For each application component from Phase C, identify which TRM service types it requires — data management, security, transaction processing, etc.

Step 2: Identify gaps and overlaps

Compare the required services against the current technology landscape (baseline). Identify where services are missing, duplicated, or where multiple competing services exist for the same need.

Step 3: Define target Technology Architecture

Using the TRM as a template and the Standards Library as a constraint, define the target set of technology platform services — selecting specific approved products for each service type.

Step 4: Apply III-RM for cross-enterprise needs

Where applications need to exchange information with external organisations, apply the III-RM to define the integration and cross-enterprise services required — API gateways, federation services, secure messaging.

⭐ Exam tip

The TRM is used in Phase D as a taxonomy and checklist for technology platform services. The III-RM is applied where cross-enterprise information sharing is required. Both are starting points that are then customised to the organisation's context using the Standards Library.
