01

Architecture Continuum — Foundation to Organisation

What the Enterprise Continuum is

The Enterprise Continuum is a model for classifying architectures and solutions from the most generic (Foundation Architecture) to the most specific (Organisation-Specific Architecture). It provides a consistent way to contextualise architecture work across industry, organisation, and project levels.

The Enterprise Continuum has two dimensions: the Architecture Continuum (for architecture descriptions and building blocks) and the Solutions Continuum (for implemented solutions). Both run from generic to specific.

Most Generic
Foundation Architecture
TOGAF Technical Reference Model, III-RM. Applies to all organisations.
↓
Common Systems Architecture
Architectures applying to multiple enterprises. E.g. federated identity, SOA frameworks.
↓
Industry Architecture
Architectures for a specific industry sector. E.g. banking, healthcare, retail reference architectures.
Most Specific
Organisation-Specific Architecture
Architectures tailored to one enterprise's needs, constraints, and context.
Direction of the Continuum

Architectures are developed from right to left — starting from generic Foundation architectures and extending towards organisation-specific ones. This prevents reinvention: architects first leverage what already exists at the foundation and common systems level before creating organisation-specific elements.

⭐ Exam tip

The Architecture Continuum runs from Foundation → Common Systems → Industry → Organisation-Specific. Foundation Architecture is the most generic (applies to all). Organisation-Specific is the most tailored. Architecture Continuum = architectural descriptions. Solutions Continuum = implementations.

02

Solutions Continuum Structure

Definition

The Solutions Continuum represents the implementations of the architectures defined in the Architecture Continuum. Where the Architecture Continuum contains ABBs, the Solutions Continuum contains SBBs — the actual products, solutions, and deployed systems.

The Solutions Continuum parallels the Architecture Continuum exactly:

Foundation Solutions
Products & Tools
COTS products, open-source tools, industry-standard components.
↓
Common Systems Solutions
Configured shared platforms, enterprise integration layers.
↓
Industry Solutions
Sector-specific packaged solutions, industry platforms.
Organisation Specific
Deployed Systems
The enterprise's specific deployed systems and custom solutions.
Architecture Continuum vs Solutions Continuum

The Architecture Continuum contains descriptions (what architecture should look like — ABBs, patterns, reference architectures). The Solutions Continuum contains implementations (what solutions actually exist — SBBs, products, deployed systems). The two continua are linked: every position in the Solutions Continuum realises the equivalent position in the Architecture Continuum.

⭐ Exam tip

Architecture Continuum = descriptions / ABBs. Solutions Continuum = implementations / SBBs. They are parallel structures at four levels: Foundation, Common Systems, Industry, Organisation-Specific.

03

Architecture Repository Components

The repository defined

The Architecture Repository is the storage mechanism for all architecture-related information produced and used during the ADM cycle. It is the enterprise's single source of truth for architecture assets — organised into six key components.

🏗
Architecture Metamodel
The enterprise's customised version of the Architecture Content Metamodel — defines what types of architectural entities and relationships are used.
🗺
Architecture Landscape
The architectural representations of the current, transition, and target states across all BDAT domains. Organised into Strategic, Segment, and Capability levels.
📚
Reference Library
Guidelines, templates, patterns, and reference architectures from external sources (TOGAF, industry standards, technology vendors) that can be leveraged.
📐
Standards Library
The approved technology products, versions, and standards the organisation uses. Constrains physical component selection in Phase D.
📋
Governance Log
Records of all governance decisions — compliance assessments, dispensations, change requests, risk acceptances. The audit trail of architecture governance.
📊
Architecture Requirements Repository
Stores all requirements — business, functional, non-functional, technical — with full traceability, version history, and status tracking.
⭐ Exam tip

Know all six Architecture Repository components: Architecture Metamodel, Architecture Landscape, Reference Library, Standards Library, Governance Log, Architecture Requirements Repository. The Standards Library constrains Phase D technology choices. The Governance Log records all governance decisions.

04

Architecture Landscape Levels

Within the Architecture Repository's Architecture Landscape, TOGAF defines three levels of abstraction for architecture descriptions. Each level has a different scope and audience.

Strategic Architecture

The highest-level summary view of the entire enterprise architecture — spanning the full BDAT stack at a high level of abstraction. Used by C-suite and senior leadership to understand the strategic direction of the enterprise architecture. Typically spans 3–5 years.

Segment Architecture

Architecture for a specific area of the enterprise — a business unit, geographic region, or functional domain. More detailed than Strategic Architecture but less detailed than Capability Architecture. Used by segment leaders and programme managers. Typically spans 1–3 years.

Capability Architecture

Architecture for a specific capability or solution — the most detailed level. Directly supports implementation projects. Used by solution architects, developers, and delivery teams. Typically spans 0–18 months.

Levels and the ADM

The three levels connect to ADM iteration patterns. Strategic Architecture is updated through full Architecture Iteration cycles. Segment Architecture is updated through Capability Iterations scoped to a business unit. Capability Architecture is produced during standard ADM cycles for specific implementation projects.

⭐ Exam tip

Three Architecture Landscape levels: Strategic (whole enterprise, C-suite, 3–5 years), Segment (business unit, programme, 1–3 years), Capability (specific solution, delivery teams, 0–18 months). Strategic is highest abstraction; Capability is lowest.

05

Reference Library, Standards Library & Governance Log

Three of the most exam-tested Architecture Repository components deserve deeper focus: the Reference Library, Standards Library, and Governance Log.

Reference Library

Stores reusable guidelines, templates, patterns, and reference architectures from external sources. These include TOGAF's own Technical Reference Model, industry reference architectures (BIAN for banking, HL7 for healthcare), technology vendor reference architectures, and internally developed architecture patterns. Teams use it to avoid reinvention.

Standards Library

Stores the enterprise's approved technology standards — the specific vendors, products, versions, and configurations the organisation has approved for use. When architects select physical technology components in Phase D, they must choose from this library. Non-standard selections require a dispensation from the Architecture Board.

Governance Log

The audit trail of all architecture governance decisions. Contains: compliance assessment outcomes (Fully Conformant / Conditionally Conformant / Non-Conformant), all dispensations granted (with rationale), architecture change request outcomes, and risk acceptance records. It provides accountability and supports post-project review and continuous improvement.

⭐ Exam tip

Reference Library = reusable patterns and external reference architectures. Standards Library = approved technology products (constrains Phase D). Governance Log = all governance decisions — compliance outcomes, dispensations, change requests. The Governance Log is the audit trail of architecture governance.
