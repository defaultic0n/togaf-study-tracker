01

Objectives of Phase A

Core objective

Phase A develops a high-level aspirational vision of the capabilities and business value to be delivered as a result of the proposed enterprise architecture. It establishes the scope, constraints, and expectations for the architecture project.

Phase A is the first phase of the ADM cycle proper. While the Preliminary Phase set up the architecture capability, Phase A is where a specific architecture engagement begins — triggered by the Request for Architecture Work from the Preliminary Phase.

Think of Phase A as the architecture equivalent of a project initiation document. It answers the critical questions that must be resolved before investing significant effort in detailed architecture work: What are we trying to achieve? Who cares? What constraints apply? Do we have the authority to proceed?

Develop the Architecture Vision

Create a high-level view of the target enterprise architecture, showing how it will deliver agreed business goals and address stakeholder concerns.

Obtain Approval

Secure formal sign-off from key stakeholders and sponsors to proceed. Without approval, no resources are committed and the ADM cycle does not advance.

Define Scope

Establish the breadth and depth of the architecture work — which domains (BDAT), which business units, what time horizons, and what is explicitly out of scope.

Identify Stakeholders

Identify everyone with an interest in the architecture and document their concerns. Stakeholder management continues throughout the ADM but is first formalised here.

Define Constraints & Assumptions

Document known constraints — time, budget, technology, regulatory — and assumptions that underpin the vision. These carry forward into all subsequent phases.

Why Phase A matters so much

Phase A sets the direction for all subsequent phases. Poor scoping here causes rework in Phases B, C, and D. Missed stakeholders cause late-stage conflicts. Vague objectives produce architectures that don't satisfy anyone. Getting Phase A right is the single greatest determinant of architecture project success.

⭐ Exam tip

Phase A is triggered by the Request for Architecture Work (from the Preliminary Phase) and produces the Statement of Architecture Work and Architecture Vision. These three documents are frequently confused in exam questions.

02

Statement of Architecture Work

Definition

The Statement of Architecture Work (SoAW) is a formal document produced during Phase A that defines the scope and approach for the architecture project. It is the contract between the architecture team and the sponsoring organisation.

The Statement of Architecture Work is one of the most important deliverables in Phase A. Once approved, it authorises the architecture team to proceed through Phases B, C, D, E, F, G, and H. Think of it as the formal project mandate for the entire architecture engagement.

📄 Statement of Architecture Work — Key Contents
  • Title and scope of the architecture work
  • Business context and business goals being addressed
  • Architecture scope — domains, time horizons, geographic scope
  • Constraints — known limitations on the architecture
  • Assumptions underpinning the scope
  • Roles and responsibilities — who does what
  • Acceptance criteria — how success will be measured
  • Schedule and budget for the architecture project
  • Risks and risk mitigation approach

The SoAW is distinct from the Request for Architecture Work that triggers Phase A. The Request comes in to Phase A as an input; the Statement of Architecture Work is produced during Phase A as an output. The Request says "we need architecture work done." The Statement says "here is exactly what work we will do and how."

Approval and governance

The Statement of Architecture Work must be reviewed and approved by key stakeholders — typically the Architecture Board and the sponsoring executive. Once approved, it becomes the authorised baseline for the architecture project. Changes to scope after approval require formal change control.

⭐ Exam tip

The SoAW is an output of Phase A (not an input). The Request for Architecture Work is the input. Never confuse these two. A common exam distractor is to offer the Statement of Architecture Work as the document that "triggers" Phase A — that is wrong; it is the Request for Architecture Work that triggers Phase A.

03

Architecture Vision Document

The Architecture Vision document is the other major output of Phase A alongside the Statement of Architecture Work. While the SoAW defines how the work will be done, the Architecture Vision defines what the target architecture will look like at a high level.

Definition

The Architecture Vision is a high-level, aspirational description of the target enterprise architecture. It communicates the intended architecture outcome to stakeholders in a way that is clear, compelling, and comprehensible — without the technical detail of later phases.

🎯 Architecture Vision — Key Contents
  • Problem description — business problem being addressed
  • Objective of the Statement of Architecture Work
  • Summary of changes to be made (high-level, not detailed)
  • Baseline Architecture (current state) — high-level summary
  • Target Architecture (future state) — high-level summary
  • Value proposition — business benefits of the target architecture
  • Risk assessment — key risks associated with the architecture
  • Constraints and assumptions
  • Stakeholder map and their key concerns

The Architecture Vision is deliberately high level. It is not a detailed technical specification — that comes in Phases B, C, and D. Its purpose is to get stakeholder alignment and approval at a strategic level before investing in detailed design.

Baseline vs Target Architecture

The Architecture Vision introduces two key concepts that run through the entire ADM. The Baseline Architecture is the current state — how things work today. The Target Architecture is the future state — how things should work after the transformation. The gap between them drives everything in Phases B through F.

Architecture Vision as a communication tool

The Architecture Vision is also a stakeholder communication document. It is written to be understood by non-technical executives and business sponsors, not just architects. It must articulate the business value of the architecture — not just the technical changes.

⭐ Exam tip

Phase A produces two key deliverables: the Statement of Architecture Work and the Architecture Vision. Know what each contains. The Vision is a high-level aspirational description of the target state — not detailed technical specs. The SoAW is the formal mandate to proceed.

04

Stakeholder Identification & Concerns

Stakeholder management is one of the most critical — and most underestimated — activities in Phase A. TOGAF places enormous emphasis on identifying the right stakeholders and understanding their specific concerns before any architecture work is done.

Why stakeholders matter so much in Phase A

A technically excellent architecture that does not address stakeholder concerns will be rejected or ignored. The purpose of architecture is to serve the enterprise — and the enterprise is made up of people with different goals, worries, and priorities. Understanding these is not optional.

Phase A formalises the process of stakeholder identification and mapping. Every major stakeholder group is identified, their concerns documented, and plans made for how to communicate with them throughout the ADM.

CEO / Executive Sponsor
Strategic alignment, business value, ROI, risk exposure, competitive advantage
CIO / IT Director
Technical feasibility, integration complexity, skills availability, budget and timeline
Business Unit Leaders
Impact on operations, change disruption, staff requirements, process implications
Enterprise Architects
Technical coherence, standards compliance, reuse of existing assets, maintainability
Security & Compliance
Regulatory compliance, data protection, security architecture, auditability
End Users
Usability, availability, performance, support and training

TOGAF introduces the concept of a Stakeholder Map — a visual tool that plots stakeholders against two dimensions: their interest in the architecture and their influence over it. High interest, high influence stakeholders require the most engagement.

Concerns drive architecture views

Each stakeholder's concerns determine which architecture views need to be produced. An executive worried about cost gets a value and cost view. A security officer worried about data protection gets a security architecture view. This is why stakeholder identification in Phase A directly shapes the work in Phases B, C, and D.

Communications Plan

A key output of stakeholder analysis is the Communications Plan — a structured plan for how the architecture team will engage with each stakeholder group throughout the ADM cycle: what information they'll receive, how frequently, and in what format.

⭐ Exam tip

Know that stakeholder concerns drive architecture views. This linkage — concern → viewpoint → view — is a fundamental TOGAF concept. Also: the Communications Plan is produced during Phase A as part of stakeholder management, not as a separate later activity.

05

Request for Architecture Work — The Trigger

Day 2 introduced the Request for Architecture Work as an output of the Preliminary Phase. Today we look at it from Phase A's perspective — as an input that formally initiates the architecture engagement.

Role in Phase A

The Request for Architecture Work is the primary input to Phase A. It tells the architecture team why architecture work has been commissioned, what business context exists, and what the sponsoring organisation expects to get from it. Without it, Phase A has no mandate to begin.

📋 Request for Architecture Work — Key Contents
  • Organisation sponsors requesting the work
  • Business context and the business problem/opportunity being addressed
  • Description of the change envisioned
  • Description of any pre-existing architectures that apply
  • Known constraints — budget, schedule, technology
  • Constraints on the architecture work itself
  • Priority of work and time constraints

The Request for Architecture Work is issued by the sponsoring organisation — typically an executive sponsor or the organisation's governance body. It represents the business's formal request for architectural guidance.

Key Inputs to Phase A
  • Request for Architecture Work ★
  • Architecture Principles (from Prelim Phase)
  • Tailored Architecture Framework
  • Architecture Repository
  • Organisational Model for EA
  • Governance and support frameworks
Key Outputs of Phase A
  • Statement of Architecture Work ★
  • Architecture Vision ★
  • Approved Statement of Architecture Work
  • Architecture Repository (updated)
  • Refined statements of business goals
  • Communications Plan
The document chain across Preliminary and Phase A

Preliminary Phase produces → Request for Architecture Work → triggers Phase A → Phase A produces → Statement of Architecture Work + Architecture Vision → authorises Phases B–H. This sequential hand-off is critical exam knowledge.

⭐ Exam tip

Master the document chain: Request for Architecture Work (Prelim output / Phase A input) → Statement of Architecture Work (Phase A output) → Architecture Vision (Phase A output). A very common exam question asks which document is an input vs output of which phase.
