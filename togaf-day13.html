01

Deliverables, Artefacts & Building Blocks

Core distinction

The Architecture Content Framework defines three types of architectural output: Deliverables (what is contractually agreed), Artefacts (what describes architecture), and Building Blocks (what implements architecture). Each has a precise meaning in TOGAF.

Deliverable
A work product contractually specified and reviewed/approved. Formally reviewed by the Architecture Board.

Examples: Architecture Definition Document, Statement of Architecture Work, Architecture Vision.
Artefact
An architectural work product that describes an aspect of the architecture. Catalogues, matrices, and diagrams are all artefacts. Artefacts are part of deliverables.

Examples: Application Interaction Matrix, Data Dissemination Diagram.
Building Block
A potentially reusable component of business, IT, or architectural capability that can be combined to deliver architectures and solutions.

Examples: CRM Service (ABB), Salesforce (SBB).
The relationship

An Architecture Definition Document (deliverable) contains multiple architecture descriptions. Each description uses catalogues, matrices, and diagrams (artefacts) to document building blocks and their relationships. Deliverables contain artefacts; artefacts describe building blocks.

⭐ Exam tip

The most tested distinction: Deliverable = contractually agreed and formally reviewed. Artefact = architectural work product describing architecture. Building Block = reusable component. A deliverable can contain artefacts; an artefact can describe building blocks. They are not interchangeable.

02

ABBs vs SBBs

Building Blocks come in two varieties: Architecture Building Blocks (ABBs) and Solution Building Blocks (SBBs). The distinction between them is one of the most frequently tested in the TOGAF Foundation exam.

Architecture Building Block (ABB)

Technology-independent. Defines what a capability does — not how it is implemented.

  • Defined during Phases B, C, D of the ADM
  • Vendor-neutral and product-agnostic
  • Lives in the Architecture Continuum
  • Example: "Customer Relationship Management capability"
  • Example: "Relational Database Management Service"
Solution Building Block (SBB)

Technology-specific. Implements an ABB with a real product or solution.

  • Identified and mapped in Phase E (Opportunities & Solutions)
  • Vendor-specific and product-named
  • Lives in the Solutions Continuum
  • Example: "Salesforce Sales Cloud" implements the CRM ABB
  • Example: "Oracle Database 21c" implements the RDBMS ABB
The mapping moment

ABBs are defined in Phases B–D as technology-independent architecture components. In Phase E — Opportunities and Solutions, ABBs are mapped to candidate SBBs for the first time. This is the first moment in the ADM when specific products and technologies are identified.

⭐ Exam tip

ABBs are defined in Phases B–D (architecture phases), are technology-independent, and live in the Architecture Continuum. SBBs are mapped in Phase E, are technology-specific products, and live in the Solutions Continuum. ABB-to-SBB mapping is a key Phase E activity.

03

The Content Metamodel

Definition

The Architecture Content Metamodel defines the types of architectural entities and the relationships between them. It provides a common vocabulary and structure for all architecture content — ensuring that architecture descriptions across phases are consistent and traceable.

The metamodel is the grammar of architecture description. Just as a grammar defines the rules for constructing sentences, the metamodel defines the rules for constructing architecture descriptions — what entities exist, what properties they have, and how they relate to each other.

Core content metamodel entities

Actor, Role, Business Service, Function, Process, Data Entity, Logical Data Component, Physical Data Component, Logical Application Component, Physical Application Component, Logical Technology Component, Physical Technology Component.

Relationships in the metamodel

Entities are connected by defined relationship types: realises (an SBB realises an ABB), supports (an application supports a business function), uses (an actor uses an application), assigns to (a role is assigned to an actor).

Traceability through the metamodel

The metamodel enables traceability from high-level business goals down to specific technology implementations. A business goal → requires capability → delivered by function → supported by application → hosted on technology component.

Extensions to the metamodel

The core metamodel can be extended to suit specific organisational needs or industry sectors. TOGAF allows enterprises to add custom entity types and relationships — provided the extension is documented and governed. This keeps the framework flexible without losing the benefits of a common vocabulary.

⭐ Exam tip

The Content Metamodel defines the types of entities and relationships used in TOGAF architecture descriptions — it is the vocabulary and grammar of architecture. It enables traceability from business goals to technology components. Know that it can be extended but must be documented.

04

Reusability of Building Blocks

Reusability is the fundamental value proposition of building blocks. TOGAF's entire Enterprise Continuum is built around the principle that well-designed architecture components should not need to be reinvented for every new architecture project.

What makes a good building block

A well-designed building block: addresses a clear business need, has a well-defined interface that hides internal complexity, is loosely coupled from other building blocks (can change internally without breaking others), has a defined specification that can be tested against, and is documented in the Architecture Repository.

Reuse across projects

A building block defined for one architecture project can be referenced by subsequent projects. Rather than redefining the CRM capability for every project, teams reference the existing CRM ABB from the Architecture Repository.

Reuse across organisations

Through the Enterprise Continuum, TOGAF-defined reference architectures (like the Technical Reference Model) provide building blocks reusable across any TOGAF-adopting organisation. Industry-standard architectures provide sector-specific reusable blocks.

Building block granularity

Building blocks can be coarse-grained (the entire CRM capability) or fine-grained (a specific authentication service). Coarser blocks are easier to reference; finer blocks offer more precise reuse opportunities. The right granularity depends on use context.

Building blocks and the Architecture Repository

All reusable building blocks are stored in the Architecture Repository — specifically in the Architecture Landscape (for organisation-specific BBs) and the Reference Library (for generic/industry BBs). Teams search the repository before creating new BBs to maximise reuse.

⭐ Exam tip

Key characteristics of a good building block: clear interface, loosely coupled, well-specified, reusable. Building blocks are stored in the Architecture Repository. The Enterprise Continuum enables reuse ranging from generic Foundation Architecture blocks to organisation-specific blocks.

05

Key Deliverables Per ADM Phase

Every ADM phase produces specific formal deliverables. These are the contractually agreed work products reviewed and approved by the Architecture Board. Knowing which deliverable belongs to which phase is core exam content.

PhasePrimary Deliverable(s)Key Artefacts produced
PreliminaryArchitecture Principles, Tailored Architecture FrameworkPrinciples catalogue, capability assessment
Phase AStatement of Architecture Work, Architecture VisionStakeholder map, value chain diagram
Phase BArchitecture Definition Document (Business chapter)Org map, Business Interaction Matrix, gap analysis
Phase CArchitecture Definition Document (IS chapter)App Interaction Matrix, Data Dissemination Diagram
Phase DArchitecture Definition Document (Technology chapter)Technology Portfolio Catalogue, Environments & Locations Diagram
Phase EArchitecture Roadmap (initial), Transition Architecture definitionsProject Context Diagram, work package definitions
Phase FImplementation & Migration Plan (final), Architecture Roadmap (final)Prioritised work package list, dependency schedule
Phase GArchitecture Contract (signed), Compliance AssessmentsGovernance Log entries, dispensation records
Phase HArchitecture Change Request records, updated Architecture Definition DocumentChange impact assessments, updated roadmap
Architecture Definition Document

The Architecture Definition Document is the master deliverable for Phases B, C, and D. It grows chapter by chapter: a Business Architecture chapter from Phase B, an Information Systems chapter from Phase C, and a Technology chapter from Phase D. By the end of Phase D it contains the complete defined architecture.

⭐ Exam tip

The Architecture Definition Document accumulates across Phases B, C, and D. The Architecture Roadmap first appears in Phase E and is finalised in Phase F. The Statement of Architecture Work is Phase A's primary output. The Architecture Contract is Phase G's primary output.
