01

Requirements Management as a Continuous Phase

Key concept

Requirements Management is a continuous process that operates throughout all ADM phases. Unlike Phases A through H, it doesn't start and stop — it runs alongside the entire architecture lifecycle, ensuring architecture requirements are captured, stored, managed, and fed into each phase as needed.

In TOGAF diagrams, Requirements Management sits at the centre of the ADM wheel — not on the perimeter with the lettered phases. This positioning is deliberate: it communicates that requirements management is not a phase you complete and move on from. It is an ongoing activity that every phase both contributes to and draws from.

Why requirements need continuous management

Requirements change. New stakeholders are identified in Phase A. The detailed architecture work in Phases B, C, and D reveals new technical requirements. Phase G implementation discovers that some requirements cannot be met as specified. Phase H identifies new business requirements from strategy changes. Without a central requirements management process, these changes would be untracked and uncoordinated.

Architecture Requirement

A quantitative statement of business need that must be met by the architecture. Requirements drive architecture decisions — each architecture element should be traceable to one or more requirements.

Requirements Repository

The store within the Architecture Repository where requirements are captured, versioned, and managed throughout their lifecycle. Provides the single source of truth for architecture requirements.

⭐ Exam tip

Requirements Management is not Phase A, not Phase B — it is a continuous process at the centre of the ADM. It feeds requirements into every phase and stores requirements from every phase. A common exam distractor suggests it's a numbered phase — it is not.

02

Requirements Impact on ADM Phases

Requirements Management works bidirectionally with every ADM phase — each phase produces requirements (feeding them into the repository) and consumes requirements (using stored requirements to guide its work).

Prelim Phase
Architecture Principles established — these become standing requirements that constrain all subsequent phases
Phase A
Stakeholder concerns captured → converted into high-level architecture requirements that scope the work
Phase B
Business requirements identified from capability gaps and value stream analysis → fed into the requirements repository
Phase C
Data and application requirements identified → functional requirements for information systems documented
Phase D
Technology and infrastructure requirements identified → technical requirements for platform services documented
Phase E–F
Requirements prioritised and assigned to work packages → traceability from requirement to implementation project
Phase G
Compliance reviews check implementations against requirements → non-conformances may update requirements
Phase H
Business and technology changes generate new requirements → update repository → may trigger ADM restart
Requirements and the Architecture Definition Document

The Architecture Requirements Specification — the document capturing all architecture requirements — feeds directly into the Architecture Definition Document. Each architecture decision in Phases B, C, and D should be traceable back to a specific requirement in this specification.

⭐ Exam tip

Requirements flow into every phase and every phase produces new requirements. The bidirectional flow between phases and the requirements repository is what makes requirements management "continuous" rather than a one-time activity.

03

Architecture Requirements Specification

Definition

The Architecture Requirements Specification is a formal document providing a quantitative statement of the business requirements, quality measures, performance criteria, and constraints that must be satisfied by the architecture.

This document is the primary output of Requirements Management in the context of architecture design. It bridges the gap between stakeholder concerns (which are qualitative — "I want the system to be fast") and architectural decisions (which must be quantitative — "response time must be under 2 seconds for 95% of transactions").

Functional Requirements
What the architecture must do — specific capabilities, behaviours, and functions it must provide.
Non-Functional Requirements
How the architecture must perform — quality attributes like performance, availability, scalability, and security.
Constraints
Limitations the architecture must operate within — budget, technology standards, regulatory compliance, timeline.
Assumptions
Conditions assumed to be true when writing requirements. If an assumption proves false, affected requirements must be revisited.
Traceability

Every architecture element — every building block, every design decision — should be traceable to a requirement. If you can't trace an architecture element to a requirement, it raises the question: why does it exist? Traceability is the mechanism that ensures the architecture stays requirements-driven rather than technically driven.

⭐ Exam tip

The Architecture Requirements Specification contains four types: functional, non-functional, constraints, and assumptions. It is the document that makes requirements actionable for architecture design — converting stakeholder concerns into quantitative, traceable specifications.

04

Storing and Managing Requirements

Requirements don't just need to be captured — they need to be actively managed throughout their lifecycle. This means versioning, prioritising, tracing, and updating them as the architecture evolves.

The Requirements Repository

Requirements are stored in the Requirements Repository — a component of the Architecture Repository. It serves as the single source of truth, ensuring all phases work from the same set of requirements and that changes are tracked.

Requirement Lifecycle

Each requirement passes through stages: Identified → Documented → Analysed → Prioritised → Assigned → Implemented → Verified → Closed. Requirements Management tracks each requirement through this lifecycle.

Prioritisation

Not all requirements can be satisfied equally — resources are finite. Prioritisation determines which requirements are addressed first, using criteria like business value, regulatory obligation, risk, and cost.

Versioning

Requirements change — new information, stakeholder feedback, or Phase G compliance findings may update them. Every version is stored so changes are auditable and the rationale for changes is documented.

Traceability Matrix

A matrix mapping requirements to the architecture elements, building blocks, and work packages that implement them. Ensures every requirement has a delivery owner and every deliverable has a justified requirement.

Requirements in the Architecture Repository

The Architecture Repository is not just for architecture artefacts — it stores the Requirements Repository, Architecture Principles, Standards Library, and Governance Log. All of these together form the complete governance and knowledge base for enterprise architecture. Requirements Management populates and draws from this central store continuously.

⭐ Exam tip

Requirements are stored in the Requirements Repository within the Architecture Repository. The Traceability Matrix links requirements to architecture elements and work packages. Both are key exam concepts.

05

Requirements and Change Control

Requirements change control is the process of managing changes to requirements in a governed way. Without it, requirements can drift — leading to architecture designs that no longer reflect what stakeholders actually need.

Requirements change control

Any change to an already-approved requirement must go through a formal change process — proposed, assessed for impact, reviewed by affected stakeholders, approved or rejected, and then updated in the Requirements Repository with version control.

How requirement changes affect the ADM

A requirement change in mid-cycle can have cascading effects. A change to a Phase A business requirement might invalidate Phase B decisions, which then cascades into Phase C and D. Requirements Management must assess this impact and trigger appropriate rework in affected phases — or raise a Change Request to Phase H if the impact is significant enough.

Key Inputs
  • Stakeholder concerns (all phases)
  • Architecture Definition Doc (B+C+D)
  • Existing requirements baseline
  • Change requests from Phase G/H
  • Business strategy and goals
Key Outputs
  • Architecture Requirements Specification ★
  • Requirements Repository (updated) ★
  • Traceability Matrix ★
  • Prioritised requirements baseline
  • Change impact assessments
Requirements Management and the Preliminary Phase

Requirements Management begins even before Phase A — in the Preliminary Phase, Architecture Principles are established. These principles function as standing requirements that constrain all subsequent phases. They are the highest-level requirements in the architecture, and all other requirements must be consistent with them.

⭐ Exam tip

Know three things about Requirements Management: (1) it is continuous — not a numbered phase; (2) it sits at the centre of the ADM wheel; (3) its primary output is the Architecture Requirements Specification feeding into the Architecture Definition Document.
