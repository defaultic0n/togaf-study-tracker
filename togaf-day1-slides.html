Section 1 of 5 — What is TOGAF?
What is TOGAF?
Definition

TOGAF — The Open Group Architecture Framework — is a globally recognised framework and methodology for developing, maintaining, and governing enterprise architectures. Owned by The Open Group.

🏛️
Framework, not a rigid standard
Organisations tailor TOGAF to their own context — never applied blindly
📅
Current version: TOGAF 9.2 (2018)
This is the version the Foundation exam is based on
🌍
Owned by The Open Group
International consortium of technology and business organisations

Section 2 of 5 — The Four Domains
The Four Architecture Domains
Remember with the acronym BDAT
🏢
Business
Strategy, processes, capabilities, governance
e.g. org charts, value streams
🗄️
Data
Logical & physical data assets, models
e.g. data entities, governance
📱
Application
Apps, interfaces, interactions with processes
e.g. ERP, CRM, app portfolio
⚙️
Technology
Hardware, networks, cloud infrastructure
e.g. servers, middleware, cloud
⭐ Exam tip

Business is about processes and capabilities. Data is about information assets. These two are commonly confused in exam questions.

Section 3 of 5 — Key Terminology
Key Terms — Enterprise & Architecture
Enterprise
Any collection of organisations sharing common goals. Not just a company — can be a government body, consortium, or whole industry sector.
Architecture
Either (1) a formal description of a system and its components, or (2) the actual structure of those components. Context determines which applies.
Enterprise Architecture
A whole-of-enterprise description covering all four BDAT domains. Provides strategic context for IT investment and change management.
Architecture Principles
General rules governing the architecture. Represent enterprise values. Example: "Data is an asset." Defined in the Preliminary Phase.
⭐ Exam tip

"Enterprise" in TOGAF is intentionally broad — always answer with "any collection of organisations sharing common goals."

Section 3 of 5 — Key Terminology
View vs Viewpoint — The Critical Distinction
Viewpoint
The template / specification for constructing a view. Reusable. Defines what to show and how to show it.
Like: the camera instructions
View
The actual output produced by applying a viewpoint to a specific architecture. Created per project.
Like: the photograph taken
Stakeholder
Individual, team, or organisation with an interest in the architecture. Includes developers, regulators, end users — not just executives.
Concern
A specific interest a stakeholder has — e.g. security, cost, performance. Architects address concerns through appropriate views.
Section 4 of 5 — The ADM Cycle
The Architecture Development Method
ADM — The Heart of TOGAF

A tested, repeatable, step-by-step process for developing enterprise architecture. Tells architects what to do, in what order, with what inputs and outputs. A cycle — not a one-time project.

🔄
10 phases in a continuous cycle
Preliminary + Phases A through H + Requirements Management at the centre
🧱
All four BDAT domains are addressed
Phases B, C, D correspond directly to Business, Information Systems, Technology
⚙️
Everything else supports the ADM
Guidelines, Repository, Governance — all feed into ADM phases
Section 4 of 5 — The ADM Cycle
ADM Phases — Prelim to Phase D
Prelim Preliminary Phase Set up architecture capability & principles
Phase A Architecture Vision Scope, stakeholders, high-level target state
Phase B Business Architecture Baseline → target business architecture
Phase C Information Systems Architecture Data + Application architectures
Phase D Technology Architecture Infrastructure supporting apps & data
Section 4 of 5 — The ADM Cycle
ADM Phases — E to Requirements Mgmt
Phase E Opportunities & Solutions Work packages, transition architectures
Phase F Migration Planning Detailed plan, priorities, costs & benefits
Phase G Implementation Governance Oversee projects, ensure conformance
Phase H Architecture Change Management Manage changes, decide on new cycle
Req. Mgmt Requirements Management Continuous — centre of the cycle
⭐ Exam tip

Requirements Management is not a numbered phase. It sits at the centre and runs continuously. This is one of the most tested ADM facts.

Section 5 of 5 — TOGAF Components
The Six TOGAF Components
Architecture Development Method
The step-by-step process. The heart of TOGAF.
ADM Guidelines & Techniques
Tools supporting specific ADM tasks: gap analysis, scenarios, risk.
Architecture Content Framework
Defines deliverables, artifacts, and building blocks.
Enterprise Continuum & Tools
Classification of assets; the Architecture Repository.
TOGAF Reference Models
TRM (technology) and III-RM (information flow).
Architecture Capability Framework
How to set up and run an EA practice in your org.

Session Summary
5 Things to Remember
1
TOGAF = framework, not standard. Owned by The Open Group. Version 9.2. Always tailored.
2
BDAT domains: Business → Data → Application → Technology. Top-down flow.
3
Term pairs: Viewpoint (template) vs View (output). Stakeholder vs Concern. Enterprise = broad.
4
ADM = 10 phases. Prelim → A → B → C → D → E → F → G → H. Requirements Mgmt at centre, always running.
5
6 components. ADM + Guidelines + Content Framework + Enterprise Continuum + Reference Models + Capability Framework.
Exam Preparation
Quick-Fire Exam Questions
❓
What does the Architecture Capability Framework provide?
→ Guidance on establishing and running an EA practice (roles, skills, governance)
❓
What is the difference between a View and a Viewpoint?
→ Viewpoint = template. View = actual output produced using that template.
❓
Which ADM phase runs continuously throughout all phases?
→ Requirements Management — it sits at the centre of the ADM cycle.
❓
What is TOGAF's definition of "enterprise"?
→ Any collection of organisations sharing a set of common goals.
