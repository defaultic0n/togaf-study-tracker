01

Phase D Overview & Objective

Objective

Phase D develops the Technology Architecture — the technology infrastructure required to support the deployment of Business, Data, and Application services. It is the "T" in BDAT and the last of the four architecture domains.

Phase D completes the BDAT architecture stack. After Phase B defined what the business does, Phase C defined what data and applications support it, Phase D now defines the technology platform on which those applications and data will run. The BDAT domains are always designed in this order — top down.

🏢
Business Architecture (Phase B)
Strategy, capabilities, value streams, processes, actors
🗄
Data Architecture (Phase C)
Data entities, logical/physical data components, data flows
📱
Application Architecture (Phase C)
Logical/physical applications, integration, portfolios
⚙️
Technology Architecture (Phase D) — TODAY
Infrastructure, platforms, networks, standards, hosting
Phase D objective in full

Develop the Baseline Technology Architecture (current state), the Target Technology Architecture (future state consistent with the Architecture Vision), perform Gap Analysis, and define candidate architecture roadmap components.

⭐ Exam tip

Phase D is the last detailed phase before Phases E and F move into delivery planning. After Phase D, the architecture (B, C, D) is complete and Phase E begins identifying how to close the gaps across all three domains.

02

Technology Components & Standards

Technology components defined

Technology Architecture identifies and categorises all the technology components needed to support the application and data layers — hardware, operating systems, middleware, network, storage, and hosting.

Technology Component

Any hardware or software element that forms part of the technology infrastructure. Can be logical (technology-independent) or physical (technology-specific). Examples: compute resource, network switch, database engine, middleware service.

Logical Technology Component

A technology capability defined independently of specific products. E.g. "Relational Database Management Service" — specifies what is needed without choosing a vendor product.

Physical Technology Component

A specific technology product. E.g. "Oracle Database 21c" implements the Relational DBMS logical component. Chosen based on standards and organisational constraints.

Technology Standards

The approved technology products, versions, and configurations the organisation uses. Defined in the Standards Library of the Architecture Repository. Standards constrain which physical technology components can be selected.

Why technology standards matter

Standards reduce operational complexity, lower support costs, enable reuse, and ensure vendor-supported products are in use. They are defined at the enterprise level — often in the Preliminary Phase — and enforced through architecture governance. Phase D must comply with them.

⭐ Exam tip

The same logical / physical distinction from Phase C applies in Phase D. Logical components are technology-independent; physical components are specific products. Design logical first, then select physical implementations that comply with standards.

03

Platform Services & Application Platform

A key concept in Phase D is the Application Platform — the shared technology layer that all application components sit on top of. Understanding this helps explain why Phase D is so important: the platform choices made here affect every application designed in Phase C.

Application Platform

The Application Platform is the complete set of technology services required to run application components. TOGAF organises these services into a taxonomy — making it clear what the technology architecture must provide.

Data Interchange Services

Services enabling applications to exchange data — messaging, queuing, API management, event streaming. The "integration bus" that connects applications.

Data Management Services

Database management, data access, data caching, and storage services. The foundation for all data persistence and retrieval.

Security Services

Authentication, authorisation, encryption, key management, audit logging. Provides the security fabric across all applications.

Operating System Services

Process management, file systems, scheduling, memory management. The fundamental layer on which all higher services run.

Network Services

Connectivity, routing, load balancing, DNS, firewall. The communication fabric enabling all distributed components to work together.

Location and Directory Services

Identity directories, service registries, service discovery. Enables components to find and connect to each other across the infrastructure.

Cloud and hybrid environments

In modern enterprises, the Application Platform may span on-premise infrastructure, private cloud, and public cloud services (IaaS, PaaS, SaaS). Phase D must model this hybrid environment — including where data resides, how applications communicate across boundaries, and what compliance constraints apply to each location.

⭐ Exam tip

The Application Platform is the shared infrastructure layer that all applications run on. Know its taxonomy of services — data interchange, data management, security, OS, network, and location/directory. The exam tests whether you understand what falls in the technology domain versus the application domain.

Technology Portfolio Catalogue & Artefacts

Phase D produces the same three artefact types as Phases B and C — Catalogues, Matrices, and Diagrams — but now focused on the technology domain. The most important is the Technology Portfolio Catalogue.

ArtefactTypeWhat it Shows
Technology Portfolio CatalogueCatalogueComplete inventory of technology components — hardware, software, middleware, cloud services — both baseline and target
Technology Standards CatalogueCatalogueApproved technology standards — approved vendors, products, versions, and configurations
Application / Technology MatrixMatrixMaps Phase C applications to the Phase D technology components they run on — the dependency map
Environments and Locations DiagramDiagramWhere technology components are deployed — data centres, cloud regions, branch offices, user locations
Platform Decomposition DiagramDiagramThe hierarchical breakdown of the Application Platform into its services and components
Processing DiagramDiagramHow processing is distributed across locations — data flows at the infrastructure level
Environments and Locations Diagram

This diagram is particularly important — it shows the physical and logical locations where technology components are deployed. For a global enterprise, this includes head office, regional offices, data centres, cloud regions, and disaster recovery sites. It directly answers the questions: where does data physically reside, and where do applications actually run?

Application / Technology Matrix

This matrix is the critical link between Phase C and Phase D — mapping each application (from Phase C) to the technology components (from Phase D) it depends on. It makes dependencies visible and is essential for impact analysis when technology changes are proposed.

⭐ Exam tip

The Technology Portfolio Catalogue and the Application / Technology Matrix are the most exam-tested Phase D artefacts. The matrix is particularly important because it links Phase C (applications) to Phase D (technology) — demonstrating how the BDAT domains connect.

05

Phase D Inputs, Outputs & Relationship to Phase C

Phase D is the final detailed architecture phase. Once complete, all four BDAT architecture domains have been addressed — and the architecture is ready for Phase E, which plans how to close the accumulated gaps.

Key Inputs
  • IS Architecture (Phase C) ★
  • Architecture Vision (Phase A)
  • Architecture Principles
  • Architecture Repository (incl. standards)
  • Existing Technology documentation
  • IT strategy and technology roadmap
Key Outputs
  • Baseline Technology Architecture ★
  • Target Technology Architecture ★
  • Technology Architecture Gap Analysis ★
  • Technology Portfolio Catalogue
  • Environments and Locations Diagram
  • Architecture Definition Doc (D chapter)
After Phase D — what happens next

After Phase D, the architecture work across all four domains is complete. Phase E takes all the gaps identified in Phase B, Phase C, and Phase D and identifies the projects, work packages, and transition architectures needed to close them. This is why Phase E is often called the "planning" phase — it turns three phases of gap analysis into an actionable delivery roadmap.

The Architecture Definition Document

By the end of Phase D, the Architecture Definition Document is substantially complete — it has accumulated a Business Architecture chapter (Phase B), an Information Systems Architecture chapter (Phase C), and now a Technology Architecture chapter (Phase D). This consolidated document becomes a key input to Phase E.

⭐ Exam tip

Phase D completes the architecture definition. After Phase D, Phases E and F focus on planning delivery, not on architecture design. The gaps from B + C + D are all consolidated in Phase E. This transition from design to delivery planning is frequently tested.
